## Summary

Six non-obvious technical findings discovered during Lab-009, each representing a gap between documentation or assumption and actual behavior in Splunk 10.2.3 on macOS (Apple Silicon, Rosetta 2).


## Finding 1:Incorrect Data Formatting Blocks Log Ingestion String

- **What Went Wrong:** The data collection system rejected every single log delivery attempt, resulting in a total connection failure (`HTTP 400 Bad Request`).

- **Why It Happened:** The data generator script sent log fields that were accidentally converted into a plain text sentence instead of remaining a structured data object. Because the data was "double-packaged," the destination system could not read it.

- **The Solution:** Adjusted the script to pass the data structure natively without pre-converting it into text.

**Fix:**

```python
# Wrong
"event": json.dumps(event_fields)

# Correct
"event": event_fields
```

**Impact:** Fixes a critical block. Without this correction, all 500 security events fail to load.

## Finding 2: sourcetype `_json` Required for Field Extraction

- **What Went Wrong:** The system successfully received the data packets, but critical security information like activity codes, IP addresses, and usernames could not be searched.

- **Why It Happened:** The system was configured to treat the incoming data as unstructured Windows text logs. Because the incoming data was actually structured JSON data, the system failed to break the text apart into separate searchable categories.
- **The Solution:** Explicitly update the system's data-type label (`sourcetype`) to `_json`

**Verification Command:**

```spl
index=compliance_demo | fieldsummary | table field, count
```

## Finding 3: Smart Mode Suppresses Field Data in Stats Queries

- **What Went Wrong:** A summary query designed to count specific security events returned only a single row with a count of `1`, despite 500 valid events existing in the database.

- **Why It Happened:** The platform defaults to "Smart Mode," a high-speed search setting. To save processing power, this mode suppresses detailed field extractions when creating mathematical summaries, making perfectly good data look missing.

- **The Solution:** Manually switch the user interface from Smart Mode to **Verbose Mode** when verifying data during testing.

---

## Finding 4: Inconsistent Number Formats Duplicate Categories

**What Went Wrong:** A query meant to group identical security activities accidentally split the same event type into two completely separate rows.

**Why It Happened:** Computers treat raw numbers (integers) and text numbers (strings) differently. The automated script sent event codes as raw numbers (e.g., `4625`), while manual command-line tests sent them as text (e.g., `"4625"`), causing the platform to treat them as separate categories.

**The Solution:** Enforce a strict text-only formatting rule for all event codes in the generation script.

```python
event_fields["EventCode"] = str(event_fields["EventCode"])
```

## Finding 5: Dashboard Field Names Must Match Script Output

**What Went Wrong:** Visual dashboard panels rendered completely empty with zero rows of data, even though verification proved the logs existed in the database.

**Why It Happened:** The user interface code was written using generic placeholder labels (like `src_ip` and `user`) that did not exactly match the specific names generated by the log script (`IpAddress` and `SubjectUserName`). The platform fails silently without an error message when a requested label does not exist.

**Correct Layout Mapping:**

| XML (wrong) | Script output (correct) |
|---|---|
| `src_ip` | `IpAddress` |
| `user` | `SubjectUserName` |
| `target_user` | `TargetUserName` |

**Verification before writing dashboard queries:**

```spl
index=compliance_demo | fieldsummary | table field, count, distinct_count
```

## Finding 6: strftime Required for Readable Timestamps in Tables

- **What Went Wrong:** The final data tables displayed activity times as unreadable 10-digit system codes (e.g., `1778796508`) instead of clear calendar dates.

- **Why It Happened:** The platform stores time internally as a continuous count of seconds (Unix epoch time). While it automatically translates this into readable dates on basic search tabs, it passes the raw numbers into summary tables unless given explicit instructions.

- **The Solution:** Insert a conversion command (`strftime`) to format the raw numbers into human-readable calendar text before rendering the table.

```spl
| eval Timestamp=strftime(_time, "%Y-%m-%d %H:%M:%S")
| table Timestamp, ...
```
